# Guide: Creating a Feature in DenoX

This walkthrough builds a complete API feature — a `notes` endpoint — the DenoX way. It is the
practical companion to the contract in [`AGENTS.md`](../../AGENTS.md); the `users` slice
([`src/api/users/`](../../src/api/users/)) is the living reference for every step.

## 0. Start with the spec (SDD)

Copy [`specs/_TEMPLATE.md`](../../specs/_TEMPLATE.md) to `specs/notes.md`, fill it in and set
`status: draft`. Implementation starts only after a maintainer flips it to `approved`. (AI agents:
stop and wait.)

## 1. Anatomy of a slice

```
src/api/notes/
├── note.model.ts        entities only (no behavior)
├── note.dto.ts          unknown → typed DTO (boundary validation)
├── note.repository.ts   interface + InMemoryNoteRepository
├── note.repository.kv.ts  Deno KV implementation (durable)
├── note.service.ts      business rules (no HTTP/HTML)
├── note.controller.ts   HTTP adapter (parse → call → envelope)
└── note.routes.ts       composition root: factory + wiring
```

Each layer only imports downward, and the service depends on the repository **interface** — that is
what lets `STORAGE_DRIVER=kv` swap persistence without touching business code.

## 2. Model and DTO

Define the entity shape in `note.model.ts`. In `note.dto.ts`, parse `unknown` into a typed DTO and
throw `ValidationException` with per-field details — those exact field names flow to the API
envelope **and** to the form helper's `[data-error-for]` slots.

## 3. Repository, service, controller

Follow the users slice: interface first, in-memory default, KV sibling if the data must survive
restarts. The service holds the rules and throws typed exceptions (`NotFoundException`,
`ConflictException`, ...); the controller only parses, calls and returns `ok(data)` — the
centralized error handler does the rest. Use the shared primitives (`logger`, `ok`/`fail`,
`escapeHtml`, `env`) — never reinvent them.

## 4. Wire it

In `note.routes.ts`, add the `createNoteRepository()` factory (memory/KV by `env.STORAGE_DRIVER`),
compose repository → service → controller and register the routes. Then one line in
`src/api/main.ts`: `registerNoteRoutes(api);`

## 5. Frontend (optional)

A page in `src/frontend/pages/notes/main.ts` gets its route generated by `deno task routes`. Forms
can talk to your new endpoint with zero JavaScript of your own — see
[`docs/form-interaction.md`](../form-interaction.md):

```html
<form data-api="/api/notes" data-target="#note-ok">...</form>
```

## 6. Tests — three layers

- `test/unit/` — DTO matrix + service with a mock from `test/mocks/`
- `test/integration/` — the wired app via `app.request()` (status, envelope)
- `test/e2e/` — real socket when the feature warrants it

## 7. Try it interactively (Insomnia)

Import [`docs/denox-insomnia.json`](../denox-insomnia.json) into Insomnia (Application → Import →
From File). It ships Local and Production environments and request examples — including deliberate
validation errors so you can see the `error.details.fields` envelope. Add your new endpoints to the
collection as part of the feature's documentation step.

## 8. Documentation and quality gate

Write `docs/notes.md` (overview, API table, errors, security), update `CHANGELOG.md`, and regenerate
the HTML API reference from the JSDoc you wrote along the way:

```bash
deno task doc     # outputs to docs/api/ (generated, git-ignored)
deno task ci      # fmt, lint, routes, type check, all tests — must be green
```

Done means: spec criteria covered by tests, `deno task ci` green, docs and changelog updated.
